## 2. Executive Summary

The domain contains a large read-only analytics library covering trade metrics, returns, drawdowns, ratios, risk, distributions, efficiency, benchmark comparison, statistical validation, a strategy-quality scorecard, and dashboard/report orchestration. Its most important working shape is the internal `overview.py` pipeline, which bypasses public tool wrappers and calls preserved raw implementation aliases such as `returns.__equity_curve_impl_impl` and `metrics.__total_trades_impl_impl`.

Three production paths invoke the overview layer: automatic simulation persistence, backtest API/result access, and saved interactive simulation sessions. These paths appear substantially implemented, but their persisted/returned shape is complicated by tool-envelope nesting. Database code contains explicit compatibility unwrapping for this shape.

The most serious defects are interface and test drift. Public metric functions return tool-envelope dictionaries, while active research and optimization callers treat them as raw numbers or tuples. Examples include `float(returns.cagr(...))`, `np.isfinite(expectancy(...))`, and tuple-unpacking `median_mae_mfe(...)`. Those call paths are therefore structurally incompatible with the current public definitions. In addition, several test modules import removed modules (`analytics.report`, `analytics.scorecard`, `analytics.trade`, `analytics.models`), and the main usage script imports a nonexistent `build_analytics_report` export.

The package facade exports 278 names, including many duplicate aliases. Most externally visible wrappers have no confirmed production caller; their underlying raw implementations are frequently used only through private double-underscore aliases inside `overview.py`. The evidence is strong for static structure and caller relationships, but runtime correctness remains unverified because the full test suite could not be executed.

Repeated per-file tool metadata constants are inventoried as grouped constants and excluded from the public-symbol count. The facade count is larger than some module counts because it publishes prefixed compatibility aliases.

## 5. Public Behaviour Inventory

### Public contract note

The public functions generated in the computation modules are AI-tool wrappers: they normalize list inputs, call preserved implementation functions, serialize pandas/NumPy results, log execution, and return dictionaries. Successful paths generally use `analytics_tool_result`; many exception paths instead return only `{'status': 'error', 'message': ...}`. Internal orchestration avoids these wrappers and calls raw `__…_impl_impl` aliases. Consequently, the same conceptual metric has two incompatible consumption forms: a public envelope and a private raw scalar/Series/dataclass.

## 6. Actual Workflows

### `V1-WF-ANALYTICS-001` — Comprehensive analytics calculation

* **Scope:** `Internal`
* **Trigger:** A caller supplies completed/open trades, initial balance, optional time bounds, and optional benchmark series.
* **Input boundary:** Trade-like objects or `DataFrame`; capital and optional benchmark/time inputs.
* **Functions and methods used:** `overview._normalize_trades` → `overview._calculate_analytics_for_subset_impl` → preserved raw implementations across returns/metrics/ratios/risks/drawdowns/distributions/efficiency/benchmark/statistical_tests → summary dictionary.
* **Files involved:** All 12 computation/orchestration files.
* **External dependencies:** pandas, NumPy, SciPy where required; optional numba.
* **Output boundary:** Category dictionaries (`metrics`, `returns`, `ratios`, `risks`, `drawdowns`, `distributions`, `efficiency`, `benchmark`, `validation`, `summary`).
* **Failure behaviour:** Raw implementation exceptions can abort the orchestration. Public wrappers catch broadly and may downgrade failures to short error dictionaries. Empty trades return empty category dictionaries.
* **Operational status:** **Partial**
* **Evidence:** `app/services/analytics/overview.py::_calculate_analytics_for_subset_impl`; calls such as `returns.__equity_curve_impl_impl`, `metrics.__total_trades_impl_impl`, `ratios.__sharpe_ratio_impl_impl`, and `risks.__value_at_risk_impl_impl`.

```text
`overview._normalize_trades`
→ `overview._calculate_analytics_for_subset_impl`
→ preserved raw implementations across returns/metrics/ratios/risks/drawdowns/distributions/efficiency/benchmark/statistical_tests
→ summary dictionary.
```

### `V1-WF-ANALYTICS-002` — Simulation completion analytics persistence

* **Scope:** `Cross-domain`
* **Trigger:** `SimulationRunner.run` completes and reporting configuration enables database persistence.

* **Functions and methods used:** `SimulationRunner._save_to_database` → `get_analytics_overview` → `build_overview_payload` → `DatabaseManager.save_backtest_snapshot`.

* **External dependencies:** Simulation result contract; pandas; SQLite manager.
* **Output boundary:** Analytics snapshot persisted with the completed backtest.
* **Failure behaviour:** Any analytics/database exception is caught by `_save_to_database`, logged, and not re-raised; the simulation may appear successful without analytics persistence.
* **Operational status:** **Partial**
* **Evidence:** `app/services/simulation/runner.py::SimulationRunner._save_to_database` imports and calls both overview functions, then saves the snapshot.

## 14. Audit Conclusions

### Valuable behaviour worth preserving

The raw calculation implementations cover a genuinely useful and broad trading-analytics surface. The highest-value behavior is the comprehensive overview workflow: trade normalization, equity/return construction, core performance metrics, drawdown/risk/ratio calculations, distribution and efficiency diagnostics, optional benchmark analysis, statistical validation, and API/dashboard payload production. This workflow is connected to simulation persistence, backtest APIs, and saved simulator sessions.


### Behaviour that exists but is disconnected

Many facade-level aggregate calculators, statistical helpers, cost-impact tools, and aliases are exported and tested or demonstrated but have no confirmed production caller. The intended agent-tool surface is especially disconnected because the package standardization function is a placeholder and the official usage script references a removed report API.


### Likely dead weight

The strongest dead-weight evidence applies to obsolete contracts rather than current computation functions: references to removed `report.py`, `scorecard.py`, `trade.py`, and `models.py`; duplicate aliases; generated self-recursive private wrappers; and tests aimed at those removed APIs. Current facade-only computation functions are not labelled dead code because external consumers and indirect tool discovery cannot be ruled out with High confidence.


### Duplicated responsibilities

Expectancy, R-multiple extraction/distribution, win-rate fraction, average loss, information ratio, equity/balance curves, CVaR/expected shortfall, and capital/notional efficiency all have duplicate implementations or aliases. The facade makes most duplicates simultaneously public, obscuring which result shape and implementation are canonical.


### Important uncertainties and manual confirmation

The central overview raw-alias pipeline appears internally coherent from static inspection, but it was not executed. Manual confirmation is required for actual test results, optional-dependency combinations, performance on realistic trade volumes, and whether any external agent/runtime imports facade symbols not visible in the repository. The current research and optimization integrations should be treated as broken until an executable run proves an unseen compatibility mechanism exists; no such mechanism was found in the inspected code.
