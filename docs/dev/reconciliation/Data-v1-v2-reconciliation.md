# Data — V1/V2 Reconciliation

## 1. Reconciliation Scope

* **Domain:** Data
* **V1 audit report:** `data-v1-audit.md` — static audit of `app/services/data` at commit `a39d26498e14772c571d75fa9a5f0e477a1dd912`.
* **V2 requirements:** `02-data.md`.
* **Comparison basis:** Capability and workflow behavior only. No code was inspected or changed during reconciliation.
* **V2 register coverage:** Every checkbox requirement, every normative Ownership/Public Capability bullet, both V2 contract matrices, the principal unboxed scope statements, and the proposed architecture prescriptions are assigned IDs and dispositions.
* **Comparison limitations:** V1 tests and live providers were not executed; live MT5/cTrader/Dukascopy/Binance behavior, deployed dynamic consumers, performance baselines, and the authoritative cross-domain response/tooling standards were unavailable.

## 2. Executive Summary

The final direction preserves the strongest demonstrated V1 value: canonical market-data retrieval, local CSV/Parquet access, SQLite cache/state, transforms, deterministic synthetic generation, and the simulation-facing tick preparation workflow. These are not retained as the existing flat package or legacy function surface; they are refactored behind focused capabilities and stable contracts.

The largest V1 corrections are mandatory. The package facade is broken, public return contracts are inconsistent, `_common.py` duplicates most of the domain, two different cache systems share names, the availability result overstates certainty, and scheduler “runs” can complete without moving data. The feed path is mock/manual rather than a real live-data workflow.

The accepted V2 additions are: an intentional JSON-safe official tool boundary, a typed internal Python data boundary, versioned canonical contracts, bounded quality reports, source readiness/capability declarations, complete cache identity/invalidation, storage manifests and path locking, real chunked backfill execution, minimum real-time feed lifecycle, workflow-aware precision at boundaries, and deterministic error/audit behavior.

Several V2 proposals are simplified or rejected. Phase 1 will not add a TSDB abstraction, select TimescaleDB/InfluxDB, build a SQLite connection pool/leak detector, compute a composite feed health score, perform hidden on-read schema migration, require multiprocessing, hand-maintain duplicate JSON schema files, or accept unmeasured performance thresholds. Automatic feed-gap backfill and public streaming remain deferred.

The recommended migration is incremental: establish contracts and the dual public/internal boundary; rebuild local sources, normalization, quality, storage, and cache; migrate proven downstream callers; then implement real jobs/backfills and the minimum feed runtime; finally promote external providers one at a time from staging. Legacy files and aliases are removed only after caller migration and parity tests.

Major open decisions concern the shared response contract, the internal DataFrame/iterator boundary, simulation ownership of tick modelling, labeling ownership, strict precision behavior across downstream domains, the minimum Phase 1 feed source, and measured benchmark/CI baselines.

## 3. Decision Principles

* Preserve behavior proven by a real runtime caller or coherent tested workflow.
* Treat V1 code as evidence, not as the future package structure or public API.
* Treat V2 as a proposal; accept only behavior supported by a workflow, safety constraint, or measurable need.
* Keep package-root official tools JSON-safe and agent-safe; keep efficient Python objects behind internal submodule boundaries.
* Prefer functions for stateless operations and classes only for state, dependencies, or lifecycle.
* Use one source of truth for timeframes, contracts, cache identity, errors, and source policy.
* Use SQLite and local files for the confirmed single-node scope; defer speculative backends and pooling.
* Fail explicitly for unavailable providers, unsupported historical calendars, stale data, unsafe paths, and strict precision failures.
* Do not silently repair gaps, switch sources, migrate schemas, or expose optimistic metadata.
* Keep one focused responsibility per file and group files by capability folders.
* Remove legacy aliases and duplicates after caller migration, not before.
* Escalate cross-domain contract and ownership questions to pipeline step 05.

## 10. Simplifications from V2

| V2 proposal | Problem | Simplified final direction |
| --- | --- | --- |
| Mandatory list of internal layers | Prescribes architecture rather than behavior and recreates V1 fragmentation | Use eight focused capability modules; private helpers receive no public requirement status. |
| `tools/data/` package and matching test paths | Contradicts repeated `app/services/data/` package requirements | Keep `app/services/data/` and aligned tests. |
| Mandatory `DataGateway` class | No demonstrated state/lifecycle need | Use thin public functions and typed internal routing functions; add a class only if lifecycle state emerges. |
| Hand-maintained JSON schema files for every contract | Creates duplicate sources of truth | Use typed contracts with stable IDs and generate schema artifacts when an external consumer needs them. |
| SQLite connection pool, idle recycling, and leak detector | Disproportionate for short-lived single-node SQLite transactions | Use short-lived connections, transactions, WAL/busy timeout as appropriate, and contention tests. |
| TSDB-ready abstraction plus TimescaleDB/InfluxDB choices | Speculative infrastructure without throughput evidence | Defer backend abstraction and vendor selection; keep state contracts backend-independent. |
| Automatic on-read schema migration | Hidden transformation and partial-data risk | Invalidate/re-ingest in Phase 1; use explicit offline migrations for persisted operational state. |
| Composite feed health score | Can hide the counters and has no validated action threshold | Expose explicit state, timestamps, capacity, counters, drift, circuit, and errors. |
| Automatic feed-gap historical backfill in Phase 1 | Requires unproven integration between feed and job runtimes | Record gap windows and expose them; add optional reconciliation after both runtimes are proven. |
| Mandatory multiprocessing | Implementation choice unsupported by benchmarks | Use bounded chunking first; isolate heavy work only after profiling. |
| Synthetic source adapter and dedicated synthetic tools | Two routes for the same local behavior | Keep dedicated generation capability; remove synthetic from external source protocol. |
| Exact limits and performance gates as immutable contracts | Numbers are not supported by V1 measurements | Use conservative configurable defaults and approve measured baselines before hard gates. |
| All metadata fields on every response | Can produce oversized, repetitive agent payloads | Define required core metadata and bounded capability-specific additions. |
| Every deterministic error repeated in many sections | Creates a large inconsistent requirements surface | Maintain one error manifest and reference it from tools/tests. |
| External providers implemented together in Phase 1 | V1 live behavior is unverified | Ship local/synthetic contracts first; keep providers staging/not-available and promote independently. |
