# Risk — V1/V2 Reconciliation

## 1. Reconciliation Scope

* **Domain:** risk
* **V1 audit report:** `docs/dev/audits/risk-v1-audit.md`
* **V2 requirements:** `05-risk.md`
* **Comparison method:** capability-to-capability reconciliation using only the two supplied documents. No code was re-inspected or modified.
* **V1 coverage:** all 30 `V1-CAP-RISK-*` capabilities and all 12 `V1-WF-RISK-*` workflows from the audit.
* **V2 coverage:** all 683 checkbox statements were assigned synthetic stable IDs in Section 6 because the source document does not yet contain a complete stable requirement catalogue. The six original `RISK-PEND-*` decisions and twelve `PERF-*` benchmark rows are also reconciled.
* **Comparison limitations:**
  * The V1 report is a static audit; tests were not executed and dynamic/external callers were not proven.
  * V2 contains repeated requirements, transition inventories, implementation prescriptions and cross-domain behaviors mixed with behavioral requirements.
  * Several V2 live-production decisions are explicitly unresolved.
  * Top-level system alignment and cross-domain ownership are intentionally deferred to pipeline step 05.
  * Cross-domain open decisions are marked for escalation; the top-level system document was not modified in this step.

## 2. Executive Summary

### Valuable V1 behaviour to preserve

The proven V1 core is the simulator and API risk pipeline: portfolio-state normalization, reproducible risk snapshots, VaR/ES and contribution calculations, exposure/concentration checks, position sizing, policy evaluation, regime detection, simulator pre-trade governance, what-if analysis, decision freshness checks, kill-switch entry blocking, and risk snapshot persistence.

### Major V1 weaknesses to correct

V1 exposes incompatible public surfaces, duplicates canonical concepts across compatibility namespaces, directly reads broker/simulator state inside Risk, uses mixed failure semantics, permits a dangerous `0.1` lot catch-all sizing fallback, maintains token replay state only in memory, hard-codes token account/broker identifiers, permits forced/manual governance outcomes, and retains stale/broken root tool examples.

### Important V2 behaviour to add

The final domain needs strict versioned Pydantic V2 contracts, Decimal/finite-value policy, injected time, explicit missing/stale evidence, pending-order exposure, account-currency conversion, one canonical `RiskDecisionPackage`, durable scoped single-use approval tokens, fixed-order decision precedence, live freshness and in-flight concurrency protection, audit hash chaining, canonical error codes, accurate official tool metadata, payload guards, and focused strategy-admission/allocation review boundaries.

### V2 complexity to reject or simplify

The reconciliation rejects Risk-owned portfolio lazy-service facades, lifecycle mutation, cost/incident/broad reporting services, execution-control mutation, and source-only requirement numbering. It merges current 50 root exports into a small official tool surface, removes a standalone decision-package construction tool, consolidates duplicate engine/model/simulation namespaces, and avoids manager/service/repository layers where stateless functions or one stateful class are sufficient.

Parametric VaR variants, full replay/timeline infrastructure, ranked recommendation engines, exit-liquidity stress, and graduated step-down controls are deferred from the initial rebuild.

### Major open decisions

The critical unresolved items are the cross-domain owner of pending-approval double-spend protection, stressed-lookback/crisis evidence ownership, Gaussian VaR waiver authority, approval-authority integration, and monthly-target accounting semantics. Exact Kelly and performance defaults remain domain decisions.

### Recommended migration direction

Refactor around one evidence-driven, deterministic Risk gate. External domains provide immutable point-in-time evidence; Risk normalizes it, calculates risk, evaluates fixed-order policy, and returns a canonical decision. Risk never fetches broker state or mutates execution, portfolio, lifecycle or incident state. Preserve proven math and workflows, replace public contracts and unsafe state/fallback behavior, merge duplicate surfaces, and defer non-essential analytical platforms.

## 3. Decision Principles

1. Preserve behavior proven by a real API, simulator, execution or persistence workflow.
2. Treat missing, stale, invalid or ambiguous safety evidence as a deterministic non-approval outcome.
3. Risk advises and gates; Execution acts; Portfolio/Data provide source evidence; Strategy/Governance own lifecycle and authorization.
4. Do not fetch broker or market state inside deterministic Risk calculations.
5. Use strict versioned public contracts and keep raw calculators private.
6. Use functions for stateless math and checks; use classes only when configuration, dependencies, durable state or lifecycle justify them.
7. Produce one canonical decision contract and one deterministic precedence/order model.
8. Merge duplicate imports and compatibility namespaces rather than preserving multiple canonical-looking APIs.
9. Keep risk-owned persistence narrow: audit facts, token state and explicitly approved decision/snapshot records.
10. Add concurrency and security controls before live promotion; defer advanced analytics that do not affect the initial safety gate.
11. Reject implementation prescriptions that belong to another domain or lack a confirmed workflow.
12. Keep one focused responsibility per file under the four-level package/module/file/public-symbol structure.

## 12. Inputs for the Final Domain README

### Approved capabilities

* **Public contracts, enums, precision, and deterministic time** — Use strict versioned Pydantic V2 public contracts, canonical decision/limit enums, Decimal rules, finite-value validation, and injected time.
* **Risk configuration and profile validation** — Load validated profile-driven config from `configs/risk/*.yaml`; require stable config hash and fail closed on missing/invalid live config.
* **Portfolio evidence normalization and risk snapshot** — Normalize explicit account, position, pending-order, strategy, symbol and bounded market evidence into immutable `PortfolioState` and reproducible `PortfolioRiskSnapshot`.
* **Exposure, concentration, and currency conversion** — Calculate symbol, strategy, currency, asset-class, direction, gross/net and correlated-cluster exposure using explicit metadata; fail closed on material conversion gaps.
* **Drawdown, loss, and prop-firm limits** — One profile-driven limit result model for daily loss, total loss, peak/current drawdown, consistency/best-day and optional monthly target tracking.
* **Margin and leverage review** — Calculate current/projected margin and effective leverage only from supplied contract, price, leverage and conversion evidence; missing material metadata blocks live review.
* **Volatility, historical VaR, and CVaR** — Use historical VaR/CVaR as the initial live method with explicit confidence, lookback, minimum observations, coverage metadata and missing-evidence outcomes.
* **Correlation and incremental portfolio risk** — Evaluate proposal interaction with current portfolio, report pair/portfolio correlation and incremental contribution, and optionally reduce size using supplied correlation evidence.
* **External market-context restrictions** — One market-context policy capability consumes external spread, slippage, session, liquidity and calendar evidence and emits deterministic limit results.
* **Position sizing recommendations** — Return `PositionSizingResult` for fixed-lot, fixed-risk, milestone, fractional Kelly, volatility and fixed-fractional sizing; never approve a trade; no non-zero fallback on failure.
* **Strategy admission review** — Review external validation/lifecycle evidence and return admission/promotion/demotion risk decisions; Strategy/Governance performs the actual transition.
* **Allocation-risk review** — Risk validates a `ProposedAllocation` against capital, lifecycle, strategy, symbol, cluster and portfolio limits; Portfolio owns source allocations and execution. Optional risk-parity suggestions remain advisory.
* **Canonical pre-trade review and decision package** — One deterministic governor validates request/state/config, checks kill switch/evidence/limits in fixed order, and returns one `RiskDecisionPackage` with primary and composite reasons.
* **Approval-token lifecycle** — Create and validate signed, scoped, expiring, revocable, single-use tokens bound to decision, action, account, strategy, symbol, config and authorized approver using durable token state.
* **Kill-switch and hard circuit-breaker review** — Risk evaluates/owns block state and recovery eligibility, always blocks live risk increase on active/unknown state, and emits revocation/audit facts; Execution/Governance enforces controls.
* **Regime assessment and limit tightening** — Assess volatility, liquidity, correlation, drawdown, crisis, news and session regimes from supplied evidence; apply deterministic profile tightening; fail closed when required evidence is absent.
* **Scenario and what-if risk analysis** — Run deterministic advisory scenario/what-if calculations from immutable evidence without changing live state; governor review is required before any action.
* **Risk decision summaries** — Generate focused Markdown/JSON risk summaries from snapshots, decisions and scenarios, clearly separating evidence, assumptions, limits, warnings and approval state.
* **Risk audit records and tamper evidence** — Emit canonical risk audit records with evidence/config/decision provenance and optional mandatory SHA-256 hash chaining through a durable storage interface.
* **Official agent-safe Risk tools** — Expose only the 12 canonical official tools as thin standard-envelope wrappers over deterministic capabilities; keep calculators/helpers private.
* **Validation and deterministic error model** — Public contracts reject invalid/non-finite data; internal domain exceptions map once at official tool/API boundaries to canonical error codes and fail-closed decisions.
* **Risk-owned durable state boundary** — Risk owns audit facts, token state and optional decision/snapshot records through narrow durable interfaces; database infrastructure and broad report storage remain external.
* **Evidence freshness, decision validity, and config compatibility** — Validate evidence `as_of`, decision expiry, material changes, clock skew and config hash before reuse; live uncertainty fails closed.
* **Concurrent approval capacity protection** — Fail closed unless one configured owner prevents double-spend via Risk reservation or Execution/Governance serialization and refreshed pending exposure.
* **Live-readiness freshness and in-flight tolerance** — External domains provide fresh portfolio/execution evidence; Risk validates freshness and bounded in-flight tolerance and returns a decision without broker access.
* **Metric and score aggregation** — Keep deterministic metrics needed by snapshots and decisions; merge score aggregation into snapshot/decision summaries unless an independent consumer is confirmed.
* **Replay and timeline reconstruction** — Keep deterministic what-if analysis; defer full replay clock/timeline/cockpit infrastructure to Simulation or a later phase.
* **Cross-domain snapshot/evidence contracts** — One versioned evidence contract family for Account, Portfolio, Position, PendingOrder, Symbol, MarketContext and governance state.
* **Security, observability, performance, and payload protection** — Apply deterministic logs/audits, accurate side-effect metadata, secret redaction, payload limits, bounded complexity, thread safety and benchmarked targets to accepted workflows.
